## ☠️ 致命傷ドクトリン TOP3（これだけは暗記）

初心者を殺すのは難しい概念じゃない。**この3つ**でほぼ全員が一度死ぬ。

> **1. 材料が合わないと、一切動かない。**
> （依存関係 / package.json）AIにいくら正しい指示を出しても、部品リストが狂っていればビルドは赤いまま。指示力の問題じゃなく材料の問題。
>
> **2. 無料サーバーは"屋台"。データは消える。**
> （永続化 / DB）Vercel等は再起動でファイルが全部飛ぶ。「保存できたのに消えた」は不具合じゃなく仕様。残したいなら外部の倉庫を借りろ。
>
> **3. 「俺の環境では動いた」は、実在する病。**
> （環境差 / .env / バージョン）自分のPCと本番は別の星。ローカルで動いた＝本番で動く、ではない。

この3つを疑う癖がつけば、生存率は跳ね上がる。

## 🔥 キラー深掘り①：依存関係（package.json / node_modules）

**何者か**：アプリが動くのに必要な「他人が作った部品」のリストと、その実体。
- `package.json`＝**買い物リスト**（何が要るか）
- `package-lock.json`＝**買ったレシートの控え**（どのバージョンを実際に入れたか）
- `node_modules`＝**買い込んだ部品の山**（超重いので GitHub には上げない＝`.gitignore`行き）

**なぜ死ぬか**：AIは平気で **古いバージョン**や**存在しないライブラリ**（ハルシネーション）を書いてくる。本番は買い物リストを見て自分で材料を揃えるので、リストが狂っていると「材料がありません」でビルドが即死する。

### 🛡️ AIが壊れた依存を出してきた時の防衛術
1. **エラー全文をそのまま貼る。** 要約厳禁。`Module not found` や `peer dependency` の文言が診断の核。
2. **指示を具体化する。** 「最新で」より事故りにくい言い方：
   - 「広く使われている**安定版（メジャーバージョン）を指定して**」
   - 「`package.json` と `package-lock.json` の**両方を見て、整合する構成に直して**」
3. **実在を疑う。** 見慣れないライブラリ名は npm で実在確認。無ければハルシネーション確定。
4. **最終手段＝クリーンインストール。** どうしても直らない時だけ `node_modules` と `package-lock.json` を消して入れ直す。
   - ※ 平常時は **lockファイルは消さずコミットする**。本番と同じ材料を再現するための"レシート"だから。消すのは壊れた時の最後の手段。

## 🔥 キラー深掘り②：データの永続化（データベース）

**何者か**：ユーザーが入れたデータを「消えないように」保存する仕組みと、その保管庫（Supabase等）。

**なぜ死ぬか（しかも気づきにくい）**：
- ローカルでは、ファイルやSQLiteに保存すると**ちゃんと残る**。だから「できた！」と思い込む。
- ところが Vercel等の本番は **使い捨ての屋台**。サーバーが再起動・スリープするたびに書き込んだファイルは**消える**。
- しかも消えるタイミングが不定なので、初心者は「保存機能が**たまにバグる**」と誤診して、原因にたどり着けないまま消耗する。

### 🛡️ 絶望する前の早期警告
- 「**ユーザーが入れたデータを、後で読み返す機能があるか？**」── YESなら、**その時点で外部DBが要る**。
- ファイル保存・ローカルDBに頼った設計は、最初から捨てる。後で移すのは地獄。
- 「とりあえずファイルに保存しといて」とAIに言われたら、それは**屋台に冷蔵庫を置こうとしている**サイン。止めろ。

> 📝 ここまでが「**消える前に気づく**」話。外部DBに移した後、「**消えた時に戻せるか（バックアップ／復旧）**」は、もう一段上の話。趣味で動かす分にはまだ先でいいが、**業務で使うなら、これを確立してから使う**。この本ではまだ扱わない（観測待ち）。

## 🔥 キラー深掘り③：環境とバージョン（.env / Node.js）

**何者か**：自分のPC（開発環境）と、世界に公開された本番環境は**別物**。設定値もエンジンの型番も違う。

**「俺のPCでは動いた」の三大原因**：
1. **環境変数が本番に無い。** `.env`（秘密のタレ）は `.gitignore` でGitHubに上がらない。→ **本番には別途、デプロイ先の画面で手入力**する必要がある。これを忘れて本番だけ動かないのが最頻出。
2. **Nodeバージョン違い。** ローカルが新しいエンジン、本番が古いエンジンだと同じコードで火が通らない。→ ローカルと本番のNodeバージョンを**明示的に揃える**。
3. **node_modules がローカルにしか無い。** ローカルで手動で入れた部品が `package.json` に書かれていないと、本番は再現できない。→ 入れた部品は必ずリストに記録。

### 🛡️ 防衛の合言葉
> 本番で死んだら、**環境変数 → Nodeバージョン → 依存リスト** の順に3点チェック。
> ローカルの成功は、本番の成功を1ミリも保証しない。

## 🧰 その他（よく出るやつ・一行集）

致命傷ではないが、知らないと地味に時間を溶かすもの。

- **ログ (Logs)**：アプリの行動履歴とエラーの叫び声。死んだらまずここ。フライトレコーダー。
- **ビルド (Build)**：開発用コードを本番用の"お弁当"に詰め直す作業。失敗原因はだいたい依存かバージョン。
- **環境変数 / .env**：パスワードやAPIキーを隠す"秘密のタレ"。本番には手で注ぎ直す。
- **CORS**：「よそ者お断り」の張り紙。フロントとバックのドメインが違うと弾かれる。
- **キャッシュ (Cache)**：気を利かせて古い弁当を出し続ける店員。更新が反映されない時の犯人。
- **ステータスコード**：404＝住所間違い／500＝厨房パンク／403＝権限なし／429＝飲みすぎ（レート制限）。
- **レート制限 / クォータ**：「1人◯回まで」。無料枠はケチですぐ枯れる。
- **タイムアウト（サーバーレス）**：本番は"出張シェフ"。「◯秒で出して帰れ」の制限で重い処理が突然死。
- **文字化け / UTF-8**：「譁・ｭ怜喧縺・」が出たら文字コードのズレ。日本語勢の地雷。
- **DNS**：ドメイン→IPの電話帳。設定後に繋がらないのは"浸透"待ち。
- **HTTPS / SSL**：鍵付き金庫で通信を送る。無いとブラウザに怒られる。
- **クローン (Clone)**：GitHubのコードを自分のPCに丸ごと持ってくる基本動作。
  ※ AIブラウザ完結やVercel直結だと、一度もcloneしない人もいる。

## 📚 応用編（生存後に読む｜隔離ゾーン）

**初期の生存には不要**。アプリが動いて、人に見せたくなってから読めばいい。

- **README**：リポジトリの看板・説明書。3ヶ月後の自分とAIへの手紙。
  - ※ ただしClaude Code等でAIに毎セッション作業させるなら、`CLAUDE.md`/READMEで前提を固めておくと**毎回の説明コストが消える**。AI委任スタイルでは"体裁"というより燃費装置。
- **システムプロンプト**：AIの"就業規則"。`CLAUDE.md`がこれ。ルールを先に固めると指示が安定する。
- **Fork**：他人のリポジトリをGitHub上に自分用コピーとして建てる。他人の作品を土台に改造する時。
- **プルリクエスト (PR)**：mainに合流していいか確認する稟議書。個人開発でも事故防止のワンクッション。
- **ブランチ / マージ**：実験は枝で、動いたら本流へ合流。
- **コミット / リバート・リセット**：コミットはセーブ"を置く"だけ。実際に巻き戻すのはrevert（履歴残す）/reset（履歴ごと消す）。
- **SSHキー / アクセストークン**：GitHubとPCを繋ぐ合言葉兼鍵。
